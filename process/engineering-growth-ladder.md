# Engineering Growth Ladder

## Why this document exists

Clear growth expectations create:

- Fair promotions
- Reduced ambiguity
- Transparent career progression
- Higher motivation

This document defines what engineering growth looks like in our iOS team.

---

# Level 1 — Junior Engineer

Focus:
- Writing correct code
- Following patterns
- Learning fundamentals

Expected capabilities:
- Implements features with guidance
- Writes basic unit tests
- Understands UIKit / SwiftUI fundamentals
- Asks good questions

Risk profile:
- Needs supervision for architecture changes

---

# Level 2 — Mid-Level Engineer

Focus:
- Autonomy within features
- Code quality ownership
- PR participation

Expected capabilities:
- Owns small features end-to-end
- Identifies simple refactoring opportunities
- Understands async/await basics
- Participates actively in code reviews

Risk profile:
- May struggle with system-wide thinking

---

# Level 3 — Senior Engineer

Focus:
- System thinking
- Concurrency safety
- Production reliability

Expected capabilities:
- Designs feature-level architecture
- Understands isolation & race conditions
- Uses Instruments for debugging
- Mentors juniors
- Anticipates failure modes

Risk profile:
- May optimize locally instead of globally

---

# Level 4 — Staff Engineer

Focus:
- Cross-team impact
- Architectural consistency
- Technical direction

Expected capabilities:
- Designs scalable systems
- Writes ADRs
- Defines technical standards
- Guides refactoring strategy
- Prevents architectural drift

Risk profile:
- Must balance idealism vs business velocity

---

# Level 5 — Principal Engineer

Focus:
- Long-term technical vision
- Organizational scale
- Risk management

Expected capabilities:
- Sets architectural direction
- Identifies systemic weaknesses
- Designs reliability strategy
- Influences product decisions
- Builds engineering culture

Risk profile:
- Must avoid over-centralization

---

# Evaluation Dimensions

Engineers are evaluated across:

- Code Quality
- Concurrency Safety
- Architecture Thinking
- Production Awareness
- Communication
- Mentorship
- Ownership
- Risk Management

Promotion is based on impact, not tenure.

---

# Leadership Principle

Promotion should answer:

“Is this person already operating at the next level consistently?”

If yes → promote.
If not → develop.
